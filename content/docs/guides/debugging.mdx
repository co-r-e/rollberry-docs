## Overview

When a capture does not produce the expected result — a blank video, missing content, or incorrect scroll behavior — Rollberry provides several tools to help you diagnose the problem. The two primary debugging mechanisms are **debug mode** (which saves individual frames as PNG images) and **JSONL logs** (which provide a structured record of every step in the capture process).

## Debug mode

Enable debug mode with the `--debug` flag to save every captured frame as an individual PNG file alongside the output video:

```bash
npx rollberry capture https://example.com --debug
```

With debug mode enabled, the output directory will contain:

```
rollberry-output/
├── capture-2026-03-19T10-30-00.mp4
├── manifest.json
├── frames/
│   ├── frame-0001.png
│   ├── frame-0002.png
│   ├── frame-0003.png
│   ├── ...
│   └── frame-0150.png
└── logs/
    └── capture.jsonl
```

The `frames/` directory contains one PNG for every frame captured during the scroll. At the default 30 FPS with a 5-second duration, this produces 150 frames.

## Examining individual frames

The saved frames let you pinpoint exactly where something went wrong:

**Check the first frame** — Open `frame-0001.png` to see what the page looked like before scrolling started. This helps identify:
- Whether the page loaded correctly
- Whether overlay elements were hidden (if using `--hide-selectors`)
- Whether the `--wait` time was sufficient for the page to render

**Check the last frame** — Open the last frame to confirm the scroll reached the bottom of the page.

**Compare frames in sequence** — Look for sudden jumps or blank sections that might indicate lazy-loading failures or scroll interruptions.

```bash
# On macOS, open all frames in Preview for quick review
open ./rollberry-output/frames/frame-*.png

# On Linux, use an image viewer
feh ./rollberry-output/frames/
```

## Reading JSONL logs

When debug mode is enabled, Rollberry writes structured logs in [JSONL format](https://jsonlines.org/) (one JSON object per line). These logs record every significant event during the capture process.

```bash
# View the raw log
cat ./rollberry-output/logs/capture.jsonl
```

Each log entry has a consistent structure:

```json
{"timestamp":"2026-03-19T10:30:00.000Z","level":"info","event":"browser_launched","details":{"headless":true}}
{"timestamp":"2026-03-19T10:30:00.500Z","level":"info","event":"page_navigated","details":{"url":"https://example.com","status":200}}
{"timestamp":"2026-03-19T10:30:01.500Z","level":"info","event":"wait_complete","details":{"waitMs":1000}}
{"timestamp":"2026-03-19T10:30:01.600Z","level":"info","event":"scroll_started","details":{"pageHeight":4200,"viewportHeight":720}}
{"timestamp":"2026-03-19T10:30:06.600Z","level":"info","event":"scroll_complete","details":{"framesCaptures":150}}
{"timestamp":"2026-03-19T10:30:07.200Z","level":"info","event":"video_encoded","details":{"outputPath":"capture-2026-03-19T10-30-00.mp4","durationSeconds":5}}
```

## Filtering logs programmatically

Since the logs are JSONL, you can use standard command-line tools to filter and analyze them:

```bash
# Show only errors
cat capture.jsonl | grep '"level":"error"'

# Show only scroll-related events
cat capture.jsonl | grep '"event":"scroll'

# Pretty-print a specific event with jq
cat capture.jsonl | jq 'select(.event == "page_navigated")'

# Extract all event names
cat capture.jsonl | jq -r '.event'
```

## Common log events

| Event | Description |
|---|---|
| `browser_launched` | Chromium browser has been started |
| `page_navigated` | Browser navigated to the target URL |
| `wait_complete` | Wait timer or selector wait finished |
| `selectors_hidden` | Overlay elements have been hidden |
| `scroll_started` | Scroll capture has begun |
| `frame_captured` | Individual frame was captured (debug only) |
| `scroll_complete` | Scroll has reached the end |
| `video_encoded` | MP4 encoding is complete |
| `capture_complete` | Entire capture process finished |
| `error` | An error occurred during capture |

## Troubleshooting with debug output

Here are some common issues and what to look for in the debug output:

**Blank or white video:**

Check `frame-0001.png`. If it is blank, the page did not load before capture started. Try increasing `--wait`:

```bash
npx rollberry capture https://example.com --wait 5000 --debug
```

Check the JSONL logs for the `page_navigated` event — confirm the HTTP status was 200.

**Content missing from the video:**

If certain sections appear blank in the frames, the content may be lazy-loaded and not triggering during the scroll. Try a slower scroll speed:

```bash
npx rollberry capture https://example.com --scroll-speed 0.5 --debug
```

**Scroll does not reach the bottom:**

Check the `scroll_started` event in the logs for the `pageHeight` value. If the page is very long, you may need a longer duration:

```bash
npx rollberry capture https://example.com --duration 20 --debug
```

**Overlay elements still visible:**

Open `frame-0001.png` and check if the overlay is present. If it is, your CSS selector may not be matching the element. Use browser DevTools to verify the correct selector, then retry:

```bash
npx rollberry capture https://example.com \
  --hide-selectors ".correct-selector" \
  --debug
```

## Performance considerations

Debug mode significantly increases disk usage and slightly slows down the capture because every frame must be written to disk as a PNG file. For a 5-second capture at 30 FPS:

- **Without debug mode:** ~1 MP4 file (a few MB)
- **With debug mode:** ~150 PNG files + 1 MP4 + JSONL logs (can be 100+ MB)

Only use `--debug` when actively troubleshooting. For production or CI/CD captures, omit the flag.

## Verbose terminal output

Even without `--debug`, Rollberry prints progress information to stderr. If you need to capture this output for later analysis:

```bash
npx rollberry capture https://example.com 2> capture.log
```

Or combine stdout and stderr:

```bash
npx rollberry capture https://example.com > capture.log 2>&1
```
